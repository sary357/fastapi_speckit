## Clarifications

### Session 2026-07-23

- Q: 本版本的讚／不讚與留言端點是否需要套用流量速率限制（rate limiting）？ → A: 是，加入以來源 IP 為基礎的速率限制（per-IP rate limiting），以符合專案憲章「所有公開端點需速率限制」之要求。
- Q: 每個來源 IP 的速率限制具體閘值為何？ → A: 每個來源 IP 每分鐘最多 3 次請求（讚／不讚與留言兩端點合計計算）。
- Q: 留言內容是否需要最大長度限制？ → A: 需要，最大長度限制為 1000 字元。
- Q: 已記錄的反應／留言資料（含來源 IP 等個資）是否需要設定保留期限或自動清除機制？ → A: 維持目前假設，長期保存、不設自動到期或清除機制；正式法遵刪除需求留待未來版本評估。

## 使用者情境與測試 *(mandatory)*

### 使用者故事 1 - 送出讚 (Like) 或不讚 (Dislike) 反應 (Priority: P1)

使用者透過 HTTP POST 請求送出「讚」或「不讚」的反應。系統收到後立即回覆成功，並在背後記錄這次反應的收到時間（UTC）與送出者的來源 IP。

**Why this priority**: 這是本功能最核心的能力，能獨立提供價值——讓系統開始收集使用者的正負向反應資料，是最小可行產品（MVP）。

**Independent Test**: 可透過送出一次「讚」與一次「不讚」的 POST 請求，驗證兩者皆能收到 HTTP 200 與 `{"status": "OK"}` 回應，並確認系統已各自記錄一筆包含 UTC 時間與來源 IP 的反應紀錄，完全獨立於留言功能即可測試。

**Acceptance Scenarios**:

1. **Given** 系統已啟動且可接受請求, **When** 使用者送出一筆「讚」的 POST 請求, **Then** 系統回應 HTTP 200 與 JSON 內容 `{"status": "OK"}`，並記錄一筆反應紀錄，內容包含收到時間（UTC）、來源 IP 與反應類型為「讚」。
2. **Given** 系統已啟動且可接受請求, **When** 使用者送出一筆「不讚」的 POST 請求, **Then** 系統回應 HTTP 200 與 JSON 內容 `{"status": "OK"}`，並記錄一筆反應紀錄，內容包含收到時間（UTC）、來源 IP 與反應類型為「不讚」。
3. **Given** 系統已啟動且可接受請求, **When** 同一個來源 IP 連續送出多筆「讚」或「不讚」, **Then** 系統對每一筆請求各自回應成功，並各自記錄為獨立的反應紀錄（不合併、不去重）。

---

### 使用者故事 2 - 送出留言 (Comment) (Priority: P2)

使用者透過 HTTP POST 請求送出留言內容。系統收到後記錄收到時間（UTC）、來源 IP 與留言內容，若處理過程完全沒有問題，回覆成功訊息。

**Why this priority**: 留言收集是第二個獨立功能，建立在與反應收集相同的基礎建設之上，但可獨立開發、測試與交付，不影響反應收集功能的運作。

**Independent Test**: 可透過送出一筆包含留言內容的 POST 請求，驗證系統回應 HTTP 200 與 `{"status": "OK"}`，並確認系統已記錄一筆包含 UTC 時間、來源 IP 與留言內容的紀錄，完全獨立於讚/不讚功能即可測試。

**Acceptance Scenarios**:

1. **Given** 系統已啟動且可接受請求, **When** 使用者送出一筆包含有效留言內容的 POST 請求, **Then** 系統回應 HTTP 200 與 JSON 內容 `{"status": "OK"}`，並記錄一筆留言紀錄，內容包含收到時間（UTC）、來源 IP 與留言文字。
2. **Given** 系統已啟動且可接受請求, **When** 使用者送出一筆缺少留言內容（例如空白或未提供）的 POST 請求, **Then** 系統拒絕此次提交、回應對應的錯誤訊息，且不會將其記錄為一筆有效留言。

---

### Edge Cases

- 當送出的請求缺少必要欄位（例如反應類型未指定，或留言內容為空白／缺漏）時，系統必須拒絕該請求並回覆錯誤訊息，不得記錄為有效資料。
- 當同一個來源 IP 在時間窗內送出的反應或留言請求超過速率限制時，系統必須拒絕超出限制的請求並回覆對應的錯誤訊息；未超過限制前，每筆請求仍各自視為獨立事件記錄，不做內容去重或合併。
- 當請求的內容格式無法解析（例如非預期的資料格式）時，系統必須回覆錯誤訊息，且不得中斷服務或記錄不完整的資料。
- 當系統暫時無法完成資料記錄（例如儲存失敗）時，系統不得回覆 `{"status": "OK"}`，避免使用者誤以為提交已成功。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統 MUST 提供一個可透過 HTTP POST 提交「讚」反應的方式。
- **FR-002**: 系統 MUST 提供一個可透過 HTTP POST 提交「不讚」反應的方式。
- **FR-003**: 當收到有效的讚／不讚提交時，系統 MUST 回應 HTTP 200 狀態碼與 JSON 內容 `{"status": "OK"}`。
- **FR-004**: 系統 MUST 為每一筆收到的讚／不讚提交記錄下列資訊：收到時間（UTC）、來源 IP 位址、反應類型（讚或不讚）。
- **FR-005**: 系統 MUST 提供一個可透過 HTTP POST 提交留言內容的方式。
- **FR-006**: 當收到有效的留言提交時，系統 MUST 回應 HTTP 200 狀態碼與 JSON 內容 `{"status": "OK"}`。
- **FR-007**: 系統 MUST 為每一筆收到的留言提交記錄下列資訊：收到時間（UTC）、來源 IP 位址、留言內容。
- **FR-008**: 系統 MUST 拒絕不符合下列規則的留言提交，並且 MUST NOT 將其記錄為有效留言：留言內容缺漏或去除前後空白後為空字串，或留言內容長度超過 1000 字元。
- **FR-009**: 系統 MUST NOT 對本版本的讚／不讚與留言提交要求任何身份驗證或授權機制。
- **FR-010**: 系統 MUST 將已記錄的讚／不讚與留言資料持久化保存，確保資料不因服務重啟而遺失。
- **FR-011**: 當請求格式無效或缺少必要資訊時，系統 MUST 回覆明確的錯誤訊息，且不得回應成功狀態。
- **FR-012**: 系統 MUST 對讚／不讚與留言兩個端點套用以來源 IP 為基礎的速率限制（per-IP rate limiting）：每個來源 IP 在任一分鐘內（讚／不讚與留言兩端點合計）最多只能送出 3 次請求；超過此限制的請求 MUST 被拒絕，並回覆對應的錯誤狀態（例如 HTTP 429），且不得記錄為有效資料。

### Key Entities *(include if feature involves data)*

- **反應紀錄 (Reaction Record)**：代表一次使用者送出的讚或不讚事件。屬性包含：反應類型（讚／不讚）、收到時間（UTC）、來源 IP 位址。
- **留言紀錄 (Comment Record)**：代表一次使用者送出的留言事件。屬性包含：留言內容（長度介於 1 至 1000 字元之間）、收到時間（UTC）、來源 IP 位址。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 使用者送出讚、不讚或留言後，能在 1 秒內收到系統的成功或錯誤回應。
- **SC-002**: 所有成功回應 `{"status": "OK"}` 的提交，其對應的資料皆會被正確保存，資料完整保存率達 100%。
- **SC-003**: 系統可同時處理至少 100 個並發提交請求，而不發生錯誤回應或資料遺失。
- **SC-004**: 每一筆已記錄的反應或留言，其 UTC 時間戳記與來源 IP 皆正確無誤，準確率達 100%。
- **SC-005**: 所有缺少必要欄位（例如空白留言）的提交，皆會被系統正確拒絕，且不會被誤記錄為有效資料，正確率達 100%。
- **SC-006**: 同一來源 IP 在 1 分鐘內送出超過 3 次提交時，第 4 次起的請求皆會被正確拒絕，拒絕正確率達 100%。
- **SC-007**: 所有超過 1000 字元長度限制的留言提交，皆會被系統正確拒絕，且不會被誤記錄為有效資料，正確率達 100%。

## Assumptions

- 本版本不需要任何使用者身份驗證或授權機制，符合使用者明確的要求。
- 讚／不讚與留言為單純的事件記錄，不與特定內容項目（例如特定文章或貼文）建立關聯；每次提交都是一筆獨立事件。
- 每次送出的讚／不讚或留言（在速率限制範圍內）皆視為新的一筆紀錄，本版本不提供合併計數、去除重複、編輯或刪除既有紀錄的機制。
- 資料將長期保存，本版本不包含自動到期、封存或清除機制。
- 來源 IP 位址以系統實際接收到請求的來源位址為準；若透過反向代理或負載平衡器轉送，如何取得原始使用者 IP 屬於實作細節，不在本規格中規範。
